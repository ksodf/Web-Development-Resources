# Contributing to Web Development Resources

Thank you for your interest in contributing! This document explains how you can help grow and improve this curated list.

## How to Contribute

### Suggesting a New Resource

1. **Check for duplicates** – Search existing entries to make sure the resource isn't already listed.
2. **Open an issue** – Use the [Issue Tracker](../../issues) to suggest a new resource before creating a pull request. Describe the resource and why it belongs in this list.
3. **Create a pull request** – Once your suggestion is acknowledged, fork the repository and submit a PR with your addition.

### Types of Contributions

- **Add a resource** – Propose a new tool, library, or learning resource.
- **Fix a link** – Report or fix a broken or outdated link.
- **Improve descriptions** – Clarify or expand existing resource descriptions.
- **Add a new category** – Suggest a new category if it doesn't fit any current one.

## Pull Request Guidelines

1. Fork the repository and create a branch from `main`.
   ```bash
   git checkout -b feature/add-<resource-name>
   ```
2. Follow the existing format when adding a resource:
   ```markdown
   - [Resource Name](https://example.com) – One-sentence description of the resource.
   ```
3. Place the resource in the correct category. If a new category is needed, add it to the **Table of Contents** as well.
4. Keep descriptions concise – one sentence is ideal.
5. Ensure the link is valid and points to the resource's official home page or documentation.
6. Submit a pull request with a descriptive title and a short explanation of what you added.

## Quality Standards

Resources added to this list should be:

- **Useful** – Genuinely helpful to web developers.
- **Accessible** – Freely available or offer a meaningful free tier.
- **Maintained** – Actively maintained and not abandoned.
- **Well-documented** – Has clear documentation or learning materials.

## Code of Conduct

By participating in this project, you agree to abide by our [Code of Conduct](CODE_OF_CONDUCT.md). Please be respectful and constructive in all interactions.

## Commit Message Guidelines

Use clear, descriptive commit messages. We follow the [Conventional Commits](https://www.conventionalcommits.org/) specification:

- `feat: add <resource-name> to CSS section`
- `fix: update broken link for <resource-name>`
- `docs: improve description for <resource-name>`
- `chore: reorganize JavaScript section`

## Getting Help

If you have any questions, feel free to [open an issue](../../issues) and we'll be happy to help.

Thank you for contributing! 🙌
